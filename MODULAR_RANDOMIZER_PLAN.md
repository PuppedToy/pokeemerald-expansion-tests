# Modular Randomizer Plan

## Context

The current randomizer is a monolithic Node.js pipeline (`puppedjs/index.js` → `writer.js`) that runs all three randomization domains (Pokédex, Trainers, Wild) in a single locked sequence. This blocks future capabilities:

- **Soul-link support**: multiple ROMs need to share the same Pokédex + Trainers but have independent Wild encounters
- **Nuzlocke re-runs**: same Pokédex + Trainers, new Wild per run — no way to do this without re-running everything
- **Web frontend**: a future site needs to send a config JSON to a server and receive artifacts back — the current monolith can't accept external config
- **Versioning & trust**: the server must be able to validate that an artifact was generated by a known version of the pipeline

The plan below: (1) adds tests first so refactoring doesn't silently break anything, (2) introduces a config system, (3) modularizes the pipeline into three composable steps, (4) documents the future frontend + backend requirements.

## Phase 0 — Testing Infrastructure ✅

Goal: make the pipeline regression-safe before touching any logic.

### 0.1 Seeded RNG — `puppedjs/rng.js`

Every `Math.random()` call in the pipeline routes through `rng.random()`.
An unseeded pipeline behaves identically (falls back to `Math.random`).
Tests call `rng.seed(42)` for determinism.

Files where `Math.random()` was replaced:
- `rebalancer.js`
- `writer.js`
- `tmRandomizer.js`
- `itemRandomizer.js`
- `teachableExpander.js`
- `rating.js`

## Phase 1 — Config System

### Design Principle

`config.json` holds **"feel" settings** — things an end user tweaks to get a different style of run, or a community bakes into a shareable config template. It is NOT a pipeline control file.

### User-Facing Config Schema

**`puppedjs/config.json`:**

```json
{
  "seed": null,
  "mode": "default",
  "difficulty": "fair",
  "rebalance": true,
  "balanceChance": 0.2
}
```

| Field | Type | Default | Source | Notes |
|-------|------|---------|--------|-------|
| `seed` | `integer \| null` | `null` | new | `null` = auto-generate + log. Critical for soul-link + reproducibility |
| `mode` | `"default" \| "nuzlocke" \| "soul-link"` | `"default"` | new | Metadata label for now; behavior differences added in future phases |
| `difficulty` | `"easy" \| "fair" \| "hard"` | `"fair"` | replaces `--difficulty=` flag | Maps to `presets.js` tier transform logic |
| `rebalance` | `boolean` | `true` | replaces `--no-balance` flag | |
| `balanceChance` | `float 0–1` | `0.2` | from `constants.js` `BALANCE_CHANCE` | Fraction of Pokémon that get rebalanced |

CLI flags remain as overrides (backwards compatibility).
`constants.js` reads `BALANCE_CHANCE` from loaded config at runtime instead of a hardcoded literal.

### Mode Semantics (current scope)

`mode` is a **metadata label** stamped on output artifacts. Future phases add behavioral rules per mode (e.g., nuzlocke encounter distribution, soul-link family exclusions across ROMs).

## Phase 2 — Pipeline Modularization

### Target Architecture

```
index.js  (thin orchestrator — ~100 lines after refactor)
  │
  ├── parser.js              ← extracted from index.js (all parseXxx functions)
  ├── config.js              ← new: loads/validates config + CLI overrides
  ├── rng.js                 ← new: seeded PRNG  ✅
  │
  ├── modules/
  │   ├── pokedexModule.js   ← Phase A: parse + rate + rebalance → pokedex artifact
  │   ├── trainersModule.js  ← Phase B: TM randomize + train teams + starters → trainers artifact
  │   └── wildModule.js      ← Phase C: wild encounters + items → wild artifact
  │
  └── writers/
      ├── sourceWriter.js    ← writes back to ROM .h/.c/.json files
      └── viewerWriter.js    ← produces out.html + viewer JS files
```

### Artifact Schemas

**`pokedex.json`** — output of Step 1:

```json
  "evoTree": { },

}
```

**`trainers.json`** — output of Step 2:

```json
  "pokedexSeed": 42,
```

### Two-Pass Teachable Expansion (critical ordering)

- **Pass 1** (pre-TM randomization): inside `pokedexModule.js`, using the current game's TM pool. Used for rating + moveset selection. Stored in `pokedex.json`.
- **Pass 2** (post-TM randomization): at the **start of** `trainersModule.js`, after `randomizeTMs()` rewrites `tms_hms.h`. Overwrites `.teachables` on pokes.
- `sourceWriter.js` uses **Trainers artifact pokes** (post-Pass-2) for writing `teachable_learnsets.h`, and **Pokedex artifact pokes** for writing `species_info/*.h`.

## Phase 3 — Future Spec

### Frontend Randomizer App

**Purpose:** Web UI that builds and sends a randomization config JSON to the server.

**Key UI flows:**
- **New Run**: mode → difficulty → rebalance toggle → seed (optional) → Generate → download artifacts / ROM patch
- **Soul-Link Session**: Player 1 generates full run → shares session URL → Players 2–N open URL → auto-populated config with shared Pokédex + Trainers artifacts → each generates only their Wild step
- **Nuzlocke Re-Run**: load existing session → regenerate only Wild step with new seed → fresh run with shared world
- **Viewer**: upload artifact bundle → renders Pokédex, Trainers, Wild tables (replaces `out.html`)

**Data sent to server** (`POST /api/randomize`):
```json
{
  "config": { "seed": 42, "mode": "soul-link", "difficulty": "fair", "rebalance": true, "balanceChance": 0.2 },
  "pokedexArtifact": null,
  "trainersArtifact": null,
  "sessionId": null
}
```

**Requirements:**
- No game source files on the frontend — all C parsing is server-side
- Artifacts are the canonical transfer format between frontend and backend
- Session storage for soul-link (server stores artifacts per session ID)
- Download button for ROM patch (server applies artifacts to base ROM)

### Backend Generator Service

**Endpoints:**
- `POST /api/randomize` — run pipeline with provided config
- `POST /api/session` — create/store a soul-link session
- `GET /api/session/:id/artifacts` — retrieve session artifacts

**Validation requirements:**
- JSON Schema validation on all `config` fields (reject unknown keys)
- `schemaVersion` check: reject artifacts below `MIN_ARTIFACT_SCHEMA_VERSION`
- Mode + steps consistency (soul-link without pokedexArtifact → reject)
- Rate limiting (pipeline is CPU-heavy: ~5–10s per full run)
- Auth: session tokens for soul-link artifact sharing

**Versioning strategy:**
- `schemaVersion` integer on every artifact (bump on breaking shape change)
- Server declares `MIN_ARTIFACT_SCHEMA_VERSION` — rejects older with clear error
- `configVersion` field for the same reason
- Seed is always logged for reproducibility + debugging
